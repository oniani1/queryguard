# v0.3 Feature Design: ignore blocks, notification channels, assertScaling

Date: 2026-04-15

## Overview

Three features for qguard v0.3, ordered by implementation sequence:

1. **ignore blocks** -- scoped suppression within a tracking context
2. **Notification channels** -- pluggable notifiers (logger, Slack, Sentry)
3. **assertScaling** -- multi-run scaling detection with populate callback

Each feature is independently shippable. No dependencies between them.

---

## Feature 1: ignore blocks

### Problem

The existing `ignore` option on `assertNoNPlusOne` accepts regex/string patterns matched against SQL. This fails when the noisy queries use the same SQL patterns as the queries you want to track. Common cases:

- Test setup/seed code fires INSERTs identical to application INSERTs
- Third-party middleware generates SELECTs you can't distinguish by SQL alone
- Known-noisy code sections produce varied queries that can't be captured by a single regex

Prosopite (`pause { }`) and nplusone (`signals.ignore`) both provide code-section suppression. Bullet does not, and users filed issues wanting it.

### API

```ts
import { ignore } from 'qguard'

await assertNoNPlusOne(async () => {
  await ignore(async () => {
    await seedTestData()       // not tracked
  })
  await handler(req, res)      // tracked
})
```

Also works inside middleware contexts:

```ts
app.get('/dashboard', qguardMiddleware(), async (req, res) => {
  await ignore(async () => {
    await loadPreferences()    // ignored
  })
  const data = await loadDashboard()  // tracked
  res.json(data)
})
```

### Implementation

Add `ignoring: boolean` field to `TrackingContext` (default `false`).

The `ignore(fn)` function:
1. Reads the current context from AsyncLocalStorage
2. If no context exists, runs `fn` directly (no-op)
3. Creates a shallow copy of the context with `ignoring: true`
4. Calls `trackingAls.run(copy, fn)`
5. ALS automatically restores the outer context when `fn` completes

In `recordQuery`, add an early return:
```ts
if (ctx.ignoring) return
```

Nesting works automatically. Inner `run()` shadows outer, restore on exit. No counter needed.

### Changes

- `src/core/tracker.ts`: add `ignoring` field to `TrackingContext`, add `ignore()` function, add early return in `recordQuery`
- `src/index.ts`: export `ignore`
- ~10 lines of new code

### Tests

- ignore block suppresses queries within it
- Queries before and after ignore block are still tracked
- Nested ignore blocks work
- ignore() outside a tracking context is a no-op
- ignore() works inside middleware tracking
- Async code inside ignore block is properly suppressed

## Feature 2: Notification channels

### Problem

qguard currently outputs detections to console only. Teams need:
- Structured logging (pino, winston) for observability pipelines
- Slack messages for team visibility
- Sentry events for production monitoring and tracking

### API

```ts
import { configure } from 'qguard'
import { loggerNotifier, slackNotifier, sentryNotifier } from 'qguard/notifiers'

configure({
  onDetection: [
    loggerNotifier(pino()),
    slackNotifier(process.env.SLACK_WEBHOOK_URL!),
    sentryNotifier(Sentry),
  ],
  notifyOnce: true,  // default: deduplicate by fingerprint hash
})
```

### Notifier type

```ts
interface NotificationContext {
  report: DetectionReport
  environment: 'test' | 'middleware'
  request?: { method: string; url: string }
  test?: { name: string; file: string }
}

type Notifier = (ctx: NotificationContext) => void | Promise<void>
```

### Built-in notifiers

**loggerNotifier(logger)**

Accepts any object with a `warn(obj, msg)` method (pino-style). Emits one structured log entry per detection with `fingerprintHash`, `normalizedSql`, `occurrences`, `totalDurationMs`, `caller`.

```ts
loggerNotifier(pino())
// winston users: loggerNotifier({ warn: (obj, msg) => winston.warn(msg, obj) })
```

**slackNotifier(webhookUrl, options?)**

Uses native `fetch()` (Node 20+ built-in) to POST to Slack webhook. Fire-and-forget. If `webhookUrl` is falsy, returns a no-op (safe in environments without the secret).

Message format: plain text with detection count, SQL fingerprint, caller frame, request URL or test name when available.

**sentryNotifier(Sentry)**

Accepts any object with `{ withScope, captureMessage }`. Emits one Sentry event per detection. Sets a custom fingerprint per qguard fingerprint hash so the same N+1 across deploys groups into one Sentry issue.

```ts
Sentry.withScope((scope) => {
  scope.setFingerprint(['qguard', detection.fingerprintHash])
  scope.setTag('qguard.occurrences', String(detection.occurrences))
  scope.setContext('qguard', { normalizedSql, caller, totalDurationMs })
  Sentry.captureMessage(`N+1: ${sql.slice(0, 80)}`, 'warning')
})
```

### Dispatch

New `dispatchNotifications(ctx, extraNotifiers?, options?)` function in `src/core/notify.ts`.

Collects `[...globalNotifiers, ...perRequestNotifiers]`. Runs through `Promise.allSettled`. On failure: `console.warn('[qguard] notifier failed: <message>')`. Never breaks tests or requests.

**Async handling:**
- Middleware: fire-and-forget (do not await)
- Test integrations: await before throwing/returning (prevents test runner exit before Slack sends)

### Deduplication

`notifyOnce: true` (default) keeps a `Set<string>` of fingerprint hashes that have already triggered notifications in the current process. Resets on `resetConfig()`.

Prevents 50 Slack messages when 50 tests hit the same N+1.

### Precedence

Global `onDetection` (from `configure()`) fires first. Per-request `onDetection` (from middleware options) fires second. Both run. Add `skipGlobalNotifiers: true` in middleware options to suppress global for a specific route.

### Verification

```ts
import { testNotifiers } from 'qguard/notifiers'
await testNotifiers()  // sends a synthetic detection through all configured notifiers
```

Builds a fake DetectionReport and dispatches it. Useful for verifying Slack webhook URLs and Sentry DSNs without triggering real N+1s.

### Changes

- `src/core/config.ts`: add `onDetection: Notifier[]` and `notifyOnce: boolean` to config
- `src/core/notify.ts` (new): `NotificationContext`, `Notifier` type, `dispatchNotifications()`
- `src/notifiers/index.ts` (new): `loggerNotifier`, `slackNotifier`, `sentryNotifier`, `testNotifiers`
- `src/integrations/shared.ts`: call `dispatchNotifications` after detection
- `src/middleware/shared.ts`: call `dispatchNotifications` with request context, update `onDetection` precedence
- `package.json`: add `qguard/notifiers` subpath export

### Tests

- Global onDetection fires on N+1 in test integration
- Global onDetection fires on N+1 in middleware
- Per-request onDetection fires alongside global
- skipGlobalNotifiers suppresses global
- notifyOnce deduplicates by fingerprint hash
- Notifier errors are caught and do not break tests
- Notifier errors are caught and do not break middleware
- slackNotifier is a no-op when URL is falsy
- testNotifiers dispatches synthetic report
- loggerNotifier calls warn with structured data
- sentryNotifier sets custom fingerprint

## Feature 3: assertScaling

### Problem

When test fixtures have only 1 record, threshold-based N+1 detection fails. The repeated query appears once, below the threshold. This is an information-theoretic limitation: with 1 record, N+1 code is indistinguishable from correct code.

Ruby's `n_plus_one_control` solves this by running the code at two data cardinalities and comparing query counts. If counts scale with data size, it's N+1. Zero false positives by definition.

A zero-config approach was evaluated and ruled out. Result row count correlation, stack frame loop detection, table name inference, and timing clustering all fail at K=1 or have unacceptable false positive rates. The populate callback is the irreducible minimum.

### API

```ts
import { assertScaling } from 'qguard/vitest'  // or qguard/jest

test('no N+1 in post list', async () => {
  await assertScaling({
    setup: async (n) => {
      await db.query('DELETE FROM posts')
      await seedPosts(n)
    },
    run: () => handler(req, res),
  })
})
```

Full options:

```ts
interface AssertScalingOptions {
  setup: (n: number) => void | Promise<void>
  run: () => void | Promise<void>
  teardown?: () => void | Promise<void>   // called before each setup
  factors?: number[]                       // default [2, 3]
  warmup?: boolean                         // default true
}
```

### Execution order

With defaults (`factors: [2, 3]`, `warmup: true`):

1. **Warmup:** `setup(2)` -> `run()` -> discard results (primes connection pools, caches)
2. **Run A:** `teardown()` if provided -> `setup(2)` -> `run()` -> collect fingerprint counts
3. **Run B:** `teardown()` if provided -> `setup(3)` -> `run()` -> collect fingerprint counts
4. **Compare:** for each fingerprint, if count at N=3 != count at N=2, flag as scaling

### Comparison algorithm

For each fingerprint hash that appears in either run:

| Count at N=2 | Count at N=3 | Verdict |
|---|---|---|
| 2 | 3 | Scaling (N+1) |
| 3 | 4 | Scaling (any growth = failure) |
| 2 | 2 | Constant (fine) |
| 0 | 3 | Scaling (conditional N+1) |
| 5 | 5 | Constant (fine) |

The question is binary: did this fingerprint's count change when data size changed? If yes, it's data-dependent. No slope calculation, no epsilon tolerance.

### Data cleanup

setup must be idempotent: truncate + seed, not append. If `setup(2)` seeds 2 records, then `setup(3)` must result in exactly 3 records (not 5). The teardown callback is an optional convenience for separating cleanup from seeding.

### Warmup

The first query on a new connection fires driver-internal queries (pg SET commands, pg_type lookups, mysql2 charset negotiation). These appear as extra fingerprints in the first measured run but not subsequent runs, causing false positives.

The warmup run executes `setup(factors[0])` -> `run()` and discards all tracking data. This primes connections, ORM query caches, and prepared statement compilations.

Disabled with `warmup: false` for users with pre-warmed connections.

### Error message

```
Query count is not constant across data sizes:

  SELECT * FROM "comments" WHERE "post_id" = ?
    N=2: 2 queries
    N=3: 3 queries
  at src/handlers/posts.ts:42
  fingerprint: a1b2c3d4

  SELECT * FROM "tags" WHERE "post_id" = ?
    N=2: 2 queries
    N=3: 3 queries
  at src/handlers/posts.ts:58
  fingerprint: e5f6a7b8
```

### Relationship with assertNoNPlusOne

Separate, complementary tools. assertNoNPlusOne is fast (1 run) and catches N+1s when data already has 2+ records. assertScaling is slower (3 runs) but catches N+1s invisible at low cardinality.

Users can compose them:

```ts
await assertScaling({
  setup: (n) => seedPosts(n),
  run: () => assertNoNPlusOne(() => handler(req, res)),
})
```

### New types

```ts
interface ScalingDetection {
  fingerprintHash: string
  normalizedSql: string
  countsPerFactor: Map<number, number>
  callerFrame: StackFrame | undefined
}

interface ScalingReport {
  scalingDetections: ScalingDetection[]
  constantFingerprints: number
  factors: number[]
}
```

### Changes

- `src/integrations/shared.ts`: add `runAssertScaling()` function
- `src/integrations/vitest.ts`: re-export `assertScaling`
- `src/integrations/jest.ts`: re-export `assertScaling`
- New types in `src/core/detector.ts` or new file

### Tests

- Scaling queries are detected (count grows with N)
- Constant queries pass (count stays the same)
- Warmup run prevents false positives from connection init
- warmup: false skips the warmup run
- Custom factors work
- teardown is called between runs
- setup is called with each factor value
- Error message shows per-fingerprint counts at each N
- Composable with assertNoNPlusOne
- Works with both pg and mysql2

## Scope exclusions

- **ORM-specific fix suggestions (issue #2):** Dropped. Bullet-style model/association suggestions require ORM-level hooks, which contradicts driver-level architecture. Prosopite proves that SQL + stack trace is sufficient. The marginal value of "Prisma: use include" on top of showing the repeated SQL and caller frame is not worth the effort.

- **Result row count tracking:** Evaluated as a zero-config alternative to assertScaling. Ruled out because row count correlation at K=1 has near-100% false positive rate. May revisit as a report enrichment feature (show parent row count for K>=2 detections) in a later version.

---

## Version and release

All three features ship as qguard v0.3.0. New subpath export `qguard/notifiers` is a non-breaking addition. The `ignore` export from `qguard` is a non-breaking addition. New config fields (`onDetection`, `notifyOnce`) have defaults that preserve existing behavior.
